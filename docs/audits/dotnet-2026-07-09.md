## Findings by area

### A — DI & composition
No findings. Every service is registered `AddSingleton`; there are **no scoped/transient services**, so classic captive-dependency (A1) cannot arise. The subagent read each singleton's fields for A3 shared-mutable-state hazards and found them all safe:
- `FeedbackStore`, `IngestService`, `CorrectionTelemetryService`, `OllamaLlmClientFactory` — stateless (state is method-local; a fresh `SqliteConnection` is opened per call).
- `ReportCache` — mutable `_entry`/`_epoch` guarded by `lock (_lock)` with a compare-and-set epoch.
- `LlmGate` / `ReportService` — only a `SemaphoreSlim` (thread-safe primitive; per-request state is locally-allocated).
- `LlmStructuringService._template` — unguarded but idempotent lazy memoization (a race only re-reads an identical string; reference write is atomic).

A4: `IConfiguration` usage is composition-root `.Bind(GetSection(...))` with `AddOptions<T>()` + `ValidateOnStart()` — proper typed-options binding, not string-key reads in business logic.

### B — EF Core & data access
No findings. **No EF Core** — data access is raw `Microsoft.Data.Sqlite` in `FeedbackStore.cs`, so B1/B2/B4 (AsNoTracking / client-side eval / DbContext concurrency) are not applicable. Adapted checks on the raw ADO.NET layer all pass:
- **Parameterization:** every command binds input via `SqliteParameter` (`$id`, `$limit`, columns). The one interpolated statement (`ALTER TABLE ... ADD COLUMN {column} {definition}` at `FeedbackStore.cs:89`) takes only compile-time-constant literals from `Initialize` — no external input, no injection surface.
- **Cancellation:** `InsertAsync`, `GetAsync`, `QueryAsync`, `CountNeedsReviewAsync` all forward the request `CancellationToken` to `OpenAsync`/`ExecuteReaderAsync`/`ReadAsync`/`ExecuteScalarAsync`/`ExecuteNonQueryAsync`.
- **Bounding:** the growing feedback table is read only through `QueryAsync`, bounded by `LIMIT $limit`; `ReportService` calls it with `opts.MaxItemsPerWindow`. The many `.ToList()` hits in `ReportService`/`CorrectionTelemetryService`/`Generator` are in-memory LINQ over already-materialized lists, correctly out of scope.

### E — error handling & resources
One low-severity finding. The **API request path is clean** — every catch in `Program.cs`, `IngestService.cs`, `ReportService.cs`, `FeedbackStore.cs` either rethrows cancellation, logs via `ILogger`, filters with `when (ex is not OperationCanceledException)`/`ct.IsCancellationRequested`, or carries an intent comment. All CLI-tool catches carry `/* best effort */` or log the exception; the typed `catch (OperationCanceledException) { break; }` sites are not catch-alls. All `using`/`await using` disposables are properly scoped (no E3).

- [tools/FeedbackIntelligence.Ctl/Board.cs:165](../../tools/FeedbackIntelligence.Ctl/Board.cs#L165) [low] — E1 swallowed-exception: `catch { return null; }` in `DatasetTag()` maps any read failure (permissions/IO, past the `File.Exists` guard) to "unknown provenance" with no log and no comment on the catch naming why.

# .NET-shape robustness audit — 2026-07-06

Read-only audit of the code changed since `dotnet-2026-07-03.md` (~10 PRs: all of
the injection hardening A1–A4, officials-intent, telemetry, the storage migration —
40 changed `.cs` files). Four concern-group auditors: DI/composition, SQLite/resource
lifecycle, async/concurrency, minimal-API surface + server-side HTML. The lens is
`.NET`-shape hazards the logic/injection reviews don't frame (lifetimes, connection
handling, concurrency, platform file semantics). **Findings apply to code the earlier
reviews passed — different lens, not a re-review.**

## Findings & disposition

| # | Sev | Finding | Disposition |
|---|-----|---------|-------------|
| 1 | **high** | **Lost-invalidation TOCTOU in the report cache.** `GenerateAsync` reads items, runs the ~20 s synthesis, then unconditionally `cache.Set`. A desk `POST /feedback` that calls `ReportCache.Invalidate()` **during** generation is clobbered by the trailing `Set`, resurrecting a stale report missing the new entry for up to `ReportCacheSeconds`. Breaks the documented centerpiece: "a fresh desk entry appears on the very next refresh." | **Fixed** — `ReportCache` now carries an invalidation epoch; `GenerateAsync` captures it before reading items and `Set` is a compare-and-set that no-ops if an ingest bumped the epoch mid-generation. Pinned by `ReportCacheTests`. |
| 2 | medium | **Non-atomic snapshot replace on Windows.** `WriteAtomicAsync` used `File.Move(overwrite:true)` = `MoveFileEx(REPLACE_EXISTING)`, which throws a sharing violation when a reader holds `report-latest.*` open (readers used `FileShare.Read`, not Delete). On the Windows demo host this races: the write side silently drops the snapshot update (caught + swallowed → stale); the read side can 500 the degraded-mode fallback endpoint. | **Fixed** — write uses `File.Replace` (NTFS replace-while-open) with a bounded retry; both readers open with `FileShare.ReadWrite\|Delete` and treat a transient `IOException` as "no snapshot" instead of a 500. |
| 3 | medium | **`GET /report` persists an arbitrary caller's window as the shared-link fallback.** `PersistSnapshotAsync` ran for **every** generated window, so a custom or near-empty window (a manager's 1-day slice, a crawler with any `?from`) overwrites the single canonical `report-latest.*` — the offline fallback then shows a misleading near-empty view. A durable state mutation on an unauthenticated GET. | **Fixed** — snapshot persistence is now an explicit **opt-in** (`GET /report?...&snapshot=true`). Only the operator's `ctl report` sets it, so `verify`/publish always get a fresh fallback; an ephemeral frontend view (any window) never clobbers it. Persistence moved to `GenerateAsync` so it applies on a cache hit too. (A first attempt — persist only when params are omitted — was reverted: every real caller sends explicit `from`/`to` even for the default, so it would have left the snapshot permanently stale; caught in review.) Pinned by `Snapshot_PersistedOnlyOnOptIn_NotOnEphemeralView`. |
| 4 | low | **Loopback rate-limit exemption trusts a forwarded-header-influenceable IP.** The exemption keys on `RemoteIpAddress` *after* `ForwardedHeaders`; if Tailscale Funnel does not append the real client IP, a remote client sending `X-Forwarded-For: 127.0.0.1` could be treated as loopback and bypass the per-IP limit. | **Documented, not changed** — contingent on Funnel's actual XFF behavior (plausible, unproven), and the fix reorders security middleware. Verify Funnel's forwarding first; if it does not append a trusted client IP, gate the exemption on the pre-forwarded connection IP (a middleware that stashes the original before `ForwardedHeaders`), or exempt only requests with no `X-Forwarded-For`. |

**DI/composition: clean** — no captive dependencies; the singletons that hold state
(`ReportCache`, `LlmGate`) are correctly lock/semaphore-synchronized; keyed-DI
Structuring/Synthesis resolution is sound.

## Note

This audit was an autonomous quality pass, not a requested change. Findings 1–3 are
fixed with regression tests (3 was first mis-fixed and reverted in review before the
correct opt-in landed — a medium finding still doesn't justify a rushed change to a
load-bearing path); finding 4 is a documented residual, premise-gated on Funnel's XFF
behavior. All changes are behind PR review — nothing merged without approval.

**Headline theme — unbounded model calls.** The two high findings and three of the mediums are one story: **no server-side timeout bounds any Ollama request.** Every model call inherits only the caller's request-abort token, and `LlmGate` holds a concurrency slot for the *entire* call, so a hung generation ties up a request thread and — with `LlmMaxConcurrency` slots exhausted — turns into a 503 for every other client. This is the robustness gap the injection/.NET audits don't frame, and it's the one worth acting on first.

## Recommended next steps

### High — do first (`fix/audit-llm-timeouts`, ~2–3 commits)
The two highs plus `LlmStructuringService.cs:33` and `ReportService.cs:426` are one coherent fix: introduce a **server-side per-call timeout** for model requests. Options — set a request timeout on the `OllamaApiClient`/`HttpClient` at `OllamaLlmClientFactory.cs:12`, and/or wrap each `GetResponseAsync` in a linked `CancellationTokenSource` with `CancelAfter(options.ModelCallTimeout)` so a stalled generation frees its `LlmGate` slot and its request thread. This closes the DoS-amplifier path at `LlmGate.cs:29` as a consequence. Add a config option (e.g. `Llm:CallTimeoutSeconds`) with a sane default.

### Medium (`fix/audit-robustness`, ~4 commits)
- `Program.cs:187` — broaden `/interpret`'s catch to mirror `/feedback` (degrade to `structure_failed` / return 503 on outage), not just `LlmBusyException`.
- `ReportService.cs:568` — give `WriteAtomicAsync` a per-writer unique temp name (e.g. include a GUID/PID) so concurrent snapshot writers don't collide; deletes the leftover-`.tmp` concern too.
- `ReportService.cs:503` — drive `AverageSeverityRank` from the active domain's severity ordering instead of a hardcoded switch, or log + flag when an out-of-taxonomy severity hits the `_ => 2` fallback.
- `ApiHost.cs:23` — `using var proc = Process.GetProcessById(pid);` (or dispose after reading) so board-render handles don't accumulate over a long `watch`.
